# Playbook: Customer Support Ticket Handling

Internal reference only — grounding for `workflow-def`'s best-practice suggestions. Never shown verbatim to the interviewee; use it to recognize which canonical step a described action maps to, and to notice gaps.

## Canonical steps

1. Ticket received (email, chat, portal, phone) and logged in a single system.
2. Automatic or manual triage: category and priority assigned.
3. Initial response sent within a defined SLA (acknowledgement, not necessarily resolution).
4. Ticket routed/assigned to the right owner or team.
5. Investigation — reproduce the issue, gather context, consult internal docs/eng if needed.
6. Customer updated on progress for anything not resolved immediately.
7. Resolution proposed and confirmed with the customer.
8. Ticket closed with a resolution code/category recorded.
9. Customer satisfaction (CSAT) survey sent post-resolution.
10. Recurring/high-impact issues escalated to product/eng as a bug or feature request.
11. Knowledge base updated if the issue is likely to recur.

## Commonly missed steps

- No SLA on first response time — customers don't know if/when they'll be heard.
- Tickets closed without a resolution code, making trend analysis impossible.
- No escalation path from support to product/eng for recurring issues.
- Knowledge base not updated after resolving a new/recurring issue, so the same investigation repeats.
- No CSAT or feedback loop captured post-resolution.

## Needed documents

- Ticket triage/priority matrix
- Response SLA policy document
- Escalation path documentation
- Knowledge base article template
- CSAT survey template
- Resolution code taxonomy
